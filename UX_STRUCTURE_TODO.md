# Divergram UX / 구조 TODO

## 배경
현재는 치명적인 고장보다는, 화면 구조와 상태 관리가 섞여 있어서 UX가 불안정해질 수 있는 구간이 보인다. 특히 모바일 네비게이션, 상태 기반 라우팅, API 기준점이 혼재되어 있어 기능이 늘수록 수정 비용이 커질 가능성이 있다.

## 확인된 구조 이슈

### 1. API 구조 기준 혼선
- 웹은 27번 API를 보는 흐름이 존재한다.
- DB는 150번 기준으로 이해되고 있다.
- 자비스와 맥 양쪽에 API가 떠 있어 구조 설명이 쉽게 꼬인다.
- 실사용 기준 API를 하나로 고정하지 않으면 개발/운영 설명이 계속 흔들릴 수 있다.

**정리 방향**
- 프론트 개발 기준 API를 하나로 고정한다.
- 예: "프론트는 항상 27번 API만 바라본다" 같은 단일 규칙 문서화.
- 보조 API나 운영 점검용 엔드포인트는 별도 문서에서만 설명한다.

### 2. CORS 설정 하드코딩
- `127.0.0.1`, `localhost`, `192.168.0.27` 같은 주소를 직접 나열하는 구조는 개발 환경이 바뀔 때마다 다시 깨질 수 있다.
- 특히 망 안에서 IP가 바뀌는 경우 반복 수정 가능성이 높다.

**정리 방향**
- 개발용 허용 규칙을 패턴 기반 또는 환경별 분리 방식으로 완화한다.
- 운영 허용 도메인과 개발 허용 origin을 분리해 관리한다.
- 문서에 "개발용 origin 추가 규칙"을 명확히 남긴다.

### 3. 모바일 상단/하단/플로팅 액션 중복
- `MobileHeader`
- `MobileFooterNav`
- 우하단 플로팅 메시지 버튼

이 세 요소가 동시에 존재하면 모바일 정보 구조가 과해지고, 시선 흐름이 분산된다.

**정리 방향**
- 하단 탭에 메시지 진입이 포함되면 플로팅 메시지 버튼은 제거하는 쪽이 자연스럽다.
- 모바일에서는 핵심 진입점 수를 줄여 Instagram 계열처럼 더 단순하게 가져간다.

### 4. 네비게이션 체계 중복
- 데스크톱 사이드바
- 모바일 전체 메뉴
- 모바일 하단 탭
- 추가 "더보기" 메뉴

이동 목적이 여러 UI에 분산되어 있어 같은 기능으로 가는 길이 너무 많다.

**정리 방향**
- 모바일 핵심 네비는 상단바 + 하단탭 위주로 단순화한다.
- 자주 쓰지 않는 항목만 더보기 메뉴로 보낸다.
- 페이지별 1차 진입 경로를 문서로 고정한다.

### 5. `App.tsx` 상태 과밀
현재 `App.tsx`에는 아래 성격이 다른 상태들이 함께 몰려 있다.

- 페이지 상태: `currentPage`, `settingsTab`
- 모달 상태: `showCreatePost`, `showSearch`, `showNotifications`, `showMessages`, `showProfileEdit`
- 선택 객체 상태: `selectedUserId`, `selectedPostId`, `selectedLocation`, `exploreTag`
- 운영 상태: `opsAuthorized`, `opsLoading`, `opsInfo`
- 신고 상태: `reportText`, `reportLoading`
- 커스텀 히스토리/제스처 상태

지금 당장은 동작하지만, 기능이 커질수록 상태 간 간섭과 회귀 버그 가능성이 커진다.

**정리 방향**
- 페이지 상태, 모달 상태, 선택 대상 상태를 분리한다.
- 가능하면 커스텀 훅이나 라우트 전용 상태 객체로 나눈다.
- `ops`, 신고, 제스처 로직은 본체에서 분리 검토.

### 6. URL 구조와 실제 라우팅 개념 혼합
- `/post`, `/profile`, `/settings` 같은 path를 쓰고 있지만,
- 실제 렌더링은 상태 기반이고,
- modal query까지 같이 섞여 있다.

이 구조는 뒤로가기, 직접 링크, 새로고침, 공유 URL에서 예상 밖 동작을 만들기 쉽다.

**정리 방향**
- 라우팅 책임을 명확히 정한다.
- 실제 URL을 유지할 거면 라우터 중심으로 간다.
- 상태 기반 렌더링을 유지할 거면 URL 반영 범위를 최소화한다.
- "페이지"와 "모달"을 URL에 어떻게 반영할지 규칙화한다.

### 7. `Feed`, `Profile` 책임 과다
- 데이터 로딩
- 정렬/추천/fallback
- 모달
- 상세 보기
- 액션 처리

이런 책임이 한 파일에 몰려 있으면 UI 디자인 수정 하나에도 영향 범위가 커진다.

**정리 방향**
- 컨테이너와 프리젠테이션 컴포넌트를 나눈다.
- 상세 보기, 액션 바, 데이터 패치 로직을 분리한다.
- 디자인 개편 전 선분리부터 하는 편이 안전하다.

### 8. 개발 서버 의존 구조
- 웹은 Vite dev server
- API는 맥 수동 실행 또는 자비스 systemd

임시 개발에는 괜찮지만, 같은 네트워크에서 여러 기기로 함께 보는 상황에는 안정성이 떨어진다.

**정리 방향**
- 프론트도 나중엔 서비스화 또는 빌드 서빙으로 정리한다.
- 최소한 "개발용", "공유 테스트용" 실행 방식을 분리 문서화한다.

### 9. 환경파일 역할 충돌 위험
- `.env.server`
- `.env.server.local`
- `.env.local`

현재는 맞춰져 있어도 운영값과 로컬값이 섞일 여지가 있다.

**정리 방향**
- 개발/운영 분리 규칙을 더 강하게 잡는다.
- 어떤 파일이 무엇의 기준인지 문서 첫머리에 못 박는다.
- 실행 스크립트가 참조하는 env 파일도 명시적으로 고정한다.
